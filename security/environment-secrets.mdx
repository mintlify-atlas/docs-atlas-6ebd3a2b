## Overview

PyWorker requires several sensitive credentials to function. These secrets must be managed properly to prevent unauthorized access, data breaches, and compromised infrastructure.

<Warning>
  **NEVER commit secrets to version control.** All sensitive credentials must be stored in environment variables only.
</Warning>

## Required Secrets

### HF_TOKEN (Hugging Face Token)

**Purpose:** Authenticates with Hugging Face Hub to download LoRA models.

**Used in:** `worker.py:214-224`

```python
if not HF_LORA_TOKEN:
    raise RuntimeError("HF_TOKEN is required to download LoRAs.")

log.info("Downloading LoRA '%s' from '%s'", lora_name, HF_LORA_REPO)
downloaded_path = hf_hub_download(
    repo_id=HF_LORA_REPO,
    filename=lora_name,
    repo_type="model",
    local_dir=str(COMFY_LORA_DIR),
    local_dir_use_symlinks=False,
    token=HF_LORA_TOKEN,
)
```

**Configuration:**

```bash
export HF_TOKEN="hf_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

**Security Notes:**
- Read-only token is sufficient for downloading models
- Token grants access to your Hugging Face account and private repositories
- Leaked tokens can be used to download your private models
- Rotate tokens immediately if exposed

**Where to obtain:**
1. Visit https://huggingface.co/settings/tokens
2. Create a new token with `read` permissions
3. Copy the token immediately (it won't be shown again)

### PYWORKER_MANIFEST_SECRET

**Purpose:** Shared secret key for HMAC-SHA256 signing and verification of LoRA manifests.

**Used in:** `worker.py:78, 158-199`

```python
MANIFEST_SECRET = os.getenv("PYWORKER_MANIFEST_SECRET", "").strip()

# Signature verification
expected_signature = hmac.new(
    MANIFEST_SECRET.encode("utf-8"),
    canonical_json(payload_to_sign).encode("utf-8"),
    hashlib.sha256,
).hexdigest()

if not hmac.compare_digest(expected_signature, signature):
    raise ValueError("lora_manifest signature mismatch.")
```

**Configuration:**

```bash
export PYWORKER_MANIFEST_SECRET="your-long-random-secret-key"
```

**Security Notes:**
- Must be identical across all PyWorker instances and the signing service
- Should be a cryptographically strong random string (minimum 32 characters)
- Changing this secret will invalidate all existing signed manifests
- Leaked secrets allow attackers to forge manifest signatures

**Generating a strong secret:**

```bash
# Linux/macOS
openssl rand -hex 32

# Python
python3 -c "import secrets; print(secrets.token_hex(32))"
```

<Warning>
  The manifest secret is the primary security control for LoRA downloads. If compromised, attackers can request any LoRA model from your workers.
</Warning>

## Environment Variable Deployment

### Vast.ai Templates

Secrets should be configured in the Vast.ai template environment variables section:

```bash
# Required
BACKEND=comfyui-json
PYWORKER_REPO=https://github.com/ByQwank/pyworker
PROVISIONING_SCRIPT=https://raw.githubusercontent.com/ByQwank/pyworker/main/default.sh
HF_TOKEN=hf_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
PYWORKER_MANIFEST_SECRET=your-long-random-secret-key

# Optional
PYWORKER_REF=main
SDK_BRANCH=main
SDK_VERSION=0.1.0
PYWORKER_REQUIRE_MANIFEST=auto
PYWORKER_MANIFEST_ENDPOINT=direct-instance
```

### Docker Containers

When running PyWorker in Docker, pass secrets via environment variables:

```bash
docker run -d \
  -e HF_TOKEN="hf_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx" \
  -e PYWORKER_MANIFEST_SECRET="your-long-random-secret-key" \
  -e BACKEND="comfyui-json" \
  your-pyworker-image:latest
```

**Never use `ENV` directives in Dockerfiles for secrets:**

```dockerfile
# ❌ WRONG - secrets baked into image layers
ENV HF_TOKEN="hf_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx"

# ✅ CORRECT - secrets passed at runtime
# (No ENV directive, passed via docker run -e or docker-compose)
```

### Docker Compose

Use environment files that are gitignored:

```yaml
# docker-compose.yml
services:
  pyworker:
    image: your-pyworker-image:latest
    env_file:
      - .env.secrets  # NOT committed to git
```

```bash
# .env.secrets (add to .gitignore)
HF_TOKEN=hf_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
PYWORKER_MANIFEST_SECRET=your-long-random-secret-key
```

## Security Best Practices

### 1. Use Secret Management Services

For production deployments, use dedicated secret management:

- **AWS Secrets Manager** - for AWS infrastructure
- **Google Secret Manager** - for GCP infrastructure
- **HashiCorp Vault** - for multi-cloud or on-premise
- **Doppler** - for application secrets across environments

### 2. Principle of Least Privilege

Grant minimum required permissions:

- Use read-only Hugging Face tokens (not write)
- Restrict API keys to specific IP ranges when possible
- Use separate secrets for development, staging, and production

### 3. Secret Rotation

Rotate secrets regularly:

```bash
# Generate new manifest secret
NEW_SECRET=$(openssl rand -hex 32)

# Update all workers and signing service
export PYWORKER_MANIFEST_SECRET="$NEW_SECRET"

# Update HF token annually or after suspected compromise
```

### 4. Audit Access

Monitor secret usage:

- Enable audit logging on Hugging Face tokens
- Track manifest signature verification failures
- Alert on multiple failed authentication attempts

### 5. Secure Communication

Always use encrypted channels:

- Use HTTPS for all API communication
- Enable TLS for internal service-to-service calls
- Never log secrets (even at debug level)

**Anti-pattern in code:**

```python
# ❌ NEVER log secrets
log.debug(f"Using HF_TOKEN: {HF_LORA_TOKEN}")
log.info(f"Manifest secret: {MANIFEST_SECRET}")

# ✅ CORRECT - log presence only
log.debug(f"HF_TOKEN configured: {bool(HF_LORA_TOKEN)}")
log.info(f"Manifest verification: {'enabled' if MANIFEST_SECRET else 'disabled'}")
```

### 6. Development vs. Production

Use different secrets for each environment:

```bash
# Development (.env.development)
PYWORKER_MANIFEST_SECRET=dev-secret-not-for-production
HF_TOKEN=hf_dev_token

# Production (environment variables in Vast.ai)
PYWORKER_MANIFEST_SECRET=prod-xxxxxxxxxxxxxxxxxxxxxxxxxx
HF_TOKEN=hf_prod_xxxxxxxxxxxxxxxxxxxxxxxxxx
```

## Incident Response

### If a Secret is Leaked

1. **Immediate Actions:**
   - Rotate the compromised secret immediately
   - Revoke the leaked token/key on the provider's platform
   - Generate a new secret using cryptographically secure methods

2. **For HF_TOKEN:**
   - Go to https://huggingface.co/settings/tokens
   - Delete the compromised token
   - Create a new token with minimum required permissions
   - Update all deployment configurations

3. **For PYWORKER_MANIFEST_SECRET:**
   - Generate new secret: `openssl rand -hex 32`
   - Update signing service configuration
   - Update all PyWorker instances
   - Monitor for signature verification failures during rollout

4. **Investigation:**
   - Check git history for the commit that exposed the secret
   - Use tools like `git-secrets` or `gitleaks` to scan repository
   - Review access logs for unauthorized usage
   - Notify security team if required by policy

### Preventing Future Leaks

**Pre-commit hooks with git-secrets:**

```bash
# Install git-secrets
git secrets --install

# Add patterns to detect
git secrets --add 'hf_[a-zA-Z0-9]{32,}'
git secrets --add 'PYWORKER_MANIFEST_SECRET="[^"]+"'
git secrets --add 'HF_TOKEN="[^"]+"'

# Scan existing history
git secrets --scan-history
```

**GitHub Secret Scanning:**

Enable secret scanning in repository settings to automatically detect leaked credentials.

## Verification Checklist

Before deploying PyWorker, verify:

- [ ] All secrets are stored in environment variables, not code
- [ ] `.env` and `.env.*` files are in `.gitignore`
- [ ] No hardcoded API keys or tokens in source files
- [ ] `PYWORKER_MANIFEST_SECRET` is at least 32 characters
- [ ] `HF_TOKEN` has minimum required permissions (read-only)
- [ ] Secrets differ between development and production
- [ ] Git history contains no exposed secrets
- [ ] Pre-commit hooks are configured to prevent secret commits
- [ ] Team members know how to rotate secrets if compromised

<Warning>
  After deployment, run `git log --all -p | grep -i 'hf_token\|manifest_secret'` to verify no secrets were accidentally committed.
</Warning>
